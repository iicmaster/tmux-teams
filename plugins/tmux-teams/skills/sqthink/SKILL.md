# Sequential Thinking Skill

เมื่อ skill นี้ถูกเรียกใช้ ให้คิดวิเคราะห์แบบ **Sequential Thinking** — คิดทีละขั้น ทบทวนได้ แตกสาขาได้ ก่อนให้คำตอบสุดท้าย

---

## Phase 1: Scoping (กำหนดขอบเขต)

ก่อนเริ่มคิด ให้ทำสิ่งนี้:

### 1. ระบุโจทย์ให้ชัด

ถามตัวเองว่าผู้ใช้ต้องการอะไร แล้วจัดประเภท:

| ประเภทงาน | ตัวอย่าง | Output template |
|---|---|---|
| **วิเคราะห์** (Analysis) | "วิเคราะห์โค้ดนี้", "ประเมินแนวทางนี้" | ใช้ template วิเคราะห์ |
| **เปรียบเทียบ** (Comparison) | "A หรือ B ดีกว่า", "เปรียบเทียบให้หน่อย" | ใช้ template เปรียบเทียบ |
| **วางแผน** (Planning) | "วางแผนให้หน่อย", "ออกแบบขั้นตอน" | ใช้ template วางแผน |
| **ตัดสินใจ** (Decision) | "ควรเลือกอะไร", "ช่วยคิดให้หน่อย" | ใช้ template ตัดสินใจ |

### 2. ตรวจสอบว่าจำเป็นต้องใช้ Sequential Thinking หรือไม่

**ข้ามกระบวนการนี้** ถ้าโจทย์:
- มีคำตอบตรงไปตรงมาคำตอบเดียว (เช่น "parse JSON นี้ให้หน่อย")
- เป็นงาน execution ไม่ใช่งานคิด (เช่น "เขียนฟังก์ชันนี้ให้" ที่ชัดเจนแล้วว่าต้องเขียนอะไร)
- ผู้ใช้ต้องการคำตอบสั้นเร็ว ไม่ได้ต้องการการวิเคราะห์เชิงลึก

ถ้าไม่แน่ใจ → ใช้ Sequential Thinking ไว้ก่อน แต่ลดจำนวนขั้นลง

### 3. รวบรวมข้อมูล

หากโจทย์ต้องการข้อมูลเพิ่ม ให้ **รวบรวมก่อนเริ่ม Phase 2** — อ่านไฟล์ที่เกี่ยวข้อง, ค้นเว็บ, หรือถามผู้ใช้ ข้อมูลต้องพร้อมก่อนเริ่มคิด ไม่ใช่ไปหาระหว่างทาง

**ตรวจสอบ `_bmad-output/` เสมอ:** ก่อนเริ่มวิเคราะห์ ให้เช็คว่าโปรเจกต์มีโฟลเดอร์ `_bmad-output/` หรือไม่ ถ้ามี ให้อ่านเอกสารที่เกี่ยวข้องในนั้นประกอบการวิเคราะห์ — โฟลเดอร์นี้มักมี game design docs, specs, และ context สำคัญของโปรเจกต์

### 4. ประเมินความซับซ้อน

กำหนดจำนวนขั้นตอนคิดเบื้องต้น:
- **งานง่าย** (เปรียบเทียบ 2 ตัวเลือกที่ชัดเจน): 3–4 ขั้น
- **งานปานกลาง** (วิเคราะห์หลายมิติ, วางแผนโปรเจกต์): 5–7 ขั้น
- **งานซับซ้อน** (ออกแบบสถาปัตยกรรม, ประเมินความเสี่ยงองค์รวม): 8–12 ขั้น

### 5. ถ้าโจทย์ไม่ชัด — ถามก่อน อย่าเดา

ตัวอย่างการถาม:
- "วิเคราะห์โค้ดนี้" → ถาม: "ต้องการวิเคราะห์ด้านไหน? เช่น performance, security, readability, maintainability?"
- "วางแผนโปรเจกต์" → ถาม: "มี deadline หรือข้อจำกัดด้าน resource ไหม?"
- "A หรือ B ดี?" → ถาม: "เกณฑ์ที่สำคัญที่สุดคืออะไร?"

## Phase 2: Sequential Thinking (คิดทีละขั้น)

ดำเนินการคิดเป็นขั้นตอนภายใน **extended thinking** โดยปฏิบัติตามกฎเหล่านี้:

### ประเภทความคิด (Thought Types)

แต่ละขั้นต้องมีป้ายกำกับ `[Thought N/~Total]` และระบุประเภท:

| Type | ใช้เมื่อ |
|---|---|
| **`[ANALYZE]`** | วิเคราะห์ข้อมูล แยกแยะปัจจัย ทำความเข้าใจมิติต่างๆ |
| **`[QUESTION]`** | ตั้งคำถามกับสมมติฐานหรือความคิดก่อนหน้า — ท้าทายตัวเอง |
| **`[REVISE → N]`** | แก้ไขขั้นที่ N เมื่อพบว่าคิดผิดหรือไม่ครบ (การทบทวนคือคุณภาพ ไม่ใช่ข้อบกพร่อง) |
| **`[BRANCH → N]`** | แตกสาขาจากขั้นที่ N เพื่อสำรวจแนวทางที่ **ต่างออกไปโดยสิ้นเชิง** (ถ้าแค่แก้ไขเล็กน้อย ใช้ REVISE แทน) |
| **`[HYPOTHESIZE]`** | ตั้งสมมติฐานหรือข้อเสนอ |
| **`[VERIFY]`** | ตรวจสอบสมมติฐานกับข้อมูล ข้อจำกัด และหลักฐาน |
| **`[CONCLUDE]`** | สรุปผล — ใช้เป็นขั้นสุดท้ายเท่านั้น |

### ลำดับการคิด

ลำดับขึ้นกับประเภทงาน แต่ทุกประเภทต้องปฏิบัติตามกฎร่วม:

**กฎร่วมทุกประเภท:**
- ขั้นแรกต้องเป็น `[ANALYZE]` เสมอ — ทำความเข้าใจโจทย์ ขอบเขต ข้อจำกัด ตัวแปร
- ต้องมี `[QUESTION]` อย่างน้อย 1 ครั้ง — ท้าทายสมมติฐานตัวเอง
- ต้องจบด้วย `[HYPOTHESIZE]` → `[VERIFY]` → `[CONCLUDE]` — ห้ามข้ามไปสรุปตรงๆ
- ปรับ `~Total` ได้ตลอด — ถ้าระหว่างคิดพบว่าต้องการขั้นเพิ่ม ให้เพิ่มได้เลย

**สำหรับงานวิเคราะห์:**
1. `[ANALYZE]` โจทย์ ขอบเขต ข้อจำกัด
2. `[ANALYZE]` แต่ละมิติ/ปัจจัยที่เกี่ยวข้อง
3. `[QUESTION]` ท้าทายสิ่งที่วิเคราะห์มา
4. `[HYPOTHESIZE]` → `[VERIFY]` → `[CONCLUDE]`

**สำหรับงานเปรียบเทียบ:**
1. `[ANALYZE]` โจทย์ + เกณฑ์การเปรียบเทียบ
2. `[ANALYZE]` ตัวเลือก A — จุดแข็ง จุดอ่อน
3. `[ANALYZE]` ตัวเลือก B (C, D...) — จุดแข็ง จุดอ่อน
4. `[QUESTION]` ท้าทายสมมติฐาน — มีอะไรที่มองข้ามไหม?
5. `[HYPOTHESIZE]` → `[VERIFY]` → `[CONCLUDE]`

**สำหรับงานวางแผน:**
1. `[ANALYZE]` เป้าหมาย ข้อจำกัด ทรัพยากร
2. `[ANALYZE]` ลำดับขั้นตอนและ dependencies
3. `[QUESTION]` มีความเสี่ยงอะไรที่ยังไม่ได้คิดถึง?
4. `[HYPOTHESIZE]` → `[VERIFY]` → `[CONCLUDE]`

**สำหรับงานตัดสินใจ:**
1. `[ANALYZE]` โจทย์ + เกณฑ์ที่สำคัญ (ถ้าผู้ใช้ไม่ระบุ ให้ถาม)
2. `[ANALYZE]` แต่ละทางเลือก
3. `[QUESTION]` สมมติฐานใดที่อาจผิด?
4. `[BRANCH → N]` ถ้ามีแนวทางที่ต่างออกไปที่ยังไม่ได้สำรวจ
5. `[HYPOTHESIZE]` → `[VERIFY]` → `[CONCLUDE]`

### กฎเพิ่มเติม

- **BRANCH vs REVISE** — ใช้ `BRANCH` เมื่อต้องการสำรวจแนวทางที่ต่างกันโดยสิ้นเชิง (เช่น "ลองคิดใหม่ถ้าเราไม่ใช้ microservices เลย?") ใช้ `REVISE` เมื่อแก้ไขข้อผิดพลาดหรือเพิ่มเติมในแนวทางเดิม
- **ข้อมูลขัดแย้ง** — ถ้าพบหลักฐานที่ขัดแย้งกัน ให้ใช้ `[QUESTION]` ตั้งคำถามว่าข้อมูลไหนน่าเชื่อถือกว่าและทำไม อย่าเลือกข้างโดยไม่ให้เหตุผล
- **ข้อมูลที่ไม่เกี่ยว ให้ตัดทิ้ง** — อย่าลากยาวเพราะข้อมูลเยอะ จับเฉพาะที่เกี่ยวข้อง
- **โจทย์หลายส่วน** — ถ้าผู้ใช้ขอ "วิเคราะห์ A แล้ววางแผน B" ให้แยกเป็นสอง sequential thinking pass ที่ต่อเนื่องกัน โดย pass แรกส่งผลสรุปเป็น input ของ pass ที่สอง

### ตัวอย่าง

```
[Thought 1/~6] [ANALYZE]
โจทย์คือการเลือก state management ระหว่าง Redux vs Zustand
ข้อจำกัด: ต้องทำงานกับทั้ง React DOM และ Phaser Canvas
เกณฑ์สำคัญ: ความง่ายในการ integrate, performance, community support

[Thought 2/~6] [ANALYZE]
Redux — middleware ecosystem ใหญ่ มี devtools ดีเยี่ยม
แต่ boilerplate เยอะ อาจเป็นภาระสำหรับโปรเจกต์ขนาดเล็ก

[Thought 3/~6] [ANALYZE]
Zustand — API เรียบง่าย subscribe ได้จากนอก React ได้ง่าย
ขนาดเล็กมาก (~1KB) ไม่มี boilerplate

[Thought 4/~7] [QUESTION]
จริงหรือที่ว่า Redux subscribe จากนอก React ได้ยาก?
→ ตรวจสอบแล้ว: store.subscribe() ทำได้ แต่ต้องเข้าถึง store instance ตรง
→ ปรับ total เป็น 7 เพราะต้องเปรียบเทียบเรื่องนี้ลึกขึ้น

[Thought 5/~7] [REVISE → 2]
แก้ไข: Redux subscribe จากนอก React ได้เช่นกัน ผ่าน store.subscribe()
แต่ Zustand ยังสะดวกกว่าเพราะ API ออกแบบมาให้ใช้นอก React ตั้งแต่แรก

[Thought 6/~7] [HYPOTHESIZE]
สมมติฐาน: Zustand เหมาะกว่าสำหรับโปรเจกต์นี้
เหตุผล: API ง่ายกว่า, ขนาดเล็ก, integrate กับ Phaser ได้ตรงไปตรงมา

[Thought 7/~7] [VERIFY]
✓ ทำงานกับ Phaser ได้ — subscribe จากนอก React ง่าย
✓ ขนาดเล็ก — ไม่เพิ่ม bundle size มาก
✓ Community — มี ecosystem เพียงพอ แม้ไม่ใหญ่เท่า Redux
✗ ข้อด้อย — devtools ไม่ดีเท่า Redux แต่ไม่ใช่ปัจจัยหลักของโปรเจกต์นี้

[CONCLUDE]
Zustand เหมาะสมที่สุด โดยมีความมั่นใจสูง (High confidence)
```

## Phase 3: Output (สรุปผลให้ผู้ใช้)

หลังจบกระบวนการคิดแล้ว ให้แสดงผลลัพธ์ตาม template ที่เหมาะกับประเภทงาน

**กฎสำคัญเรื่องภาษา:** ใช้ภาษาเดียวกับที่ผู้ใช้พิมพ์มา ถ้าผู้ใช้พิมพ์ไทยตอบไทย ถ้าพิมพ์อังกฤษตอบอังกฤษ ถ้าผสม ให้ใช้ไทยเป็นหลัก

**กฎสำคัญเรื่องความกระชับ:** ปรับความละเอียดตามความซับซ้อนของโจทย์ — งานง่ายไม่จำเป็นต้องแสดง section ทุกอัน ตัด section ที่ว่างเปล่าออก

## Rules (สรุปกฎทั้งหมด)

1. **ห้ามตอบตรงๆ โดยไม่ผ่าน Sequential Thinking** — ยกเว้นงานที่ไม่ต้องการการวิเคราะห์ (ดู Phase 1 ข้อ 2)
2. **ใช้ภาษาเดียวกับผู้ใช้** — ภาษาระหว่างคิดจะเป็นอะไรก็ได้ แต่ output ต้องตรงกับภาษาผู้ใช้
3. **อ้างอิงข้อมูลจริง** — อย่าสร้างข้อมูลขึ้นมาเอง ถ้าไม่รู้ให้บอกว่าไม่รู้
4. **กระชับ** — ผลลัพธ์สุดท้ายต้องตรงประเด็น ไม่ยืดยาว ตัด section ที่ว่างเปล่าออก
5. **แสดง confidence level เสมอ** — สะท้อนว่าข้อมูลที่มีเพียงพอแค่ไหน
6. **รวบรวมข้อมูลก่อนคิด** — ห้ามเริ่ม Phase 2 โดยยังขาดข้อมูลสำคัญ
